    🌟 프로젝트 기여

  • EC2 Linux 환경에서 Jenkins, GitLab, Docker를 활용한 CI/CD 파이프라인 구축
  • 외부 API를 활용한 환율 정보 조회 및 환전 API 구현

    • 🛠️ 사용 기술

    Spring Boot, Docker, Redis, Jenkins, GitLab, AWS EC2, Ubuntu, Spring Data JPA, MySQL

    📝 사용 이유

  • Docker: 일관된 환경에서 애플리케이션을 배포하기 위해 사용했습니다.
  • Redis: 자주 조회되는 환율 정보를 빠르게 전달하기 위해 사용했습니다.
  • Lua Script: 환율을 갱신하는 과정에서 삭제와 삽입 연산을 하나의 트랜잭션으로 묶어 원자성을 보장하기 위해 사용했습니다.
  • Firebase Cloud Messaging (FCM) : 실시간 PUSH 알림 기능을 안정적이고 손쉽게 구현하기 위해 사용했습니다.

    • ⚠️ 발생한 문제

    외부 API를 통해 환율 데이터를 받아와 Redis에 저장하는 과정에서, Redis가 단일 스레드 기반으로 작동하여 개별 명령어의 원자성은 보장되지만, 복수의 명령어를 순차적으로 실행할 때 완전한 원자성이 보장되지 않을 수 있다는 점을 인식했습니다. 우리가 만드는 프로젝트가 핀테크 서비스를 다루고 있어 결제, 송금, 충전 등 실제 돈이 오가는 시스템이기 때문에, 정확한 환율 데이터를 제공하는 것이 매우 중요하다고 생각했습니다. 이런 특성상, 조금의 오류 가능성도 배제할 수 없다고 생각했습니다.

    💡 시도한 방법

    Lua Script를 사용하면 Script에 작성한 연산들은 하나의 트랜잭션으로 관리된다는 것을 검색을 통해 알게 됐고 환율 삭제와 삽입 명령어를 Lua Script로 작성해 하나의 트랜잭션으로 묶어 데이터 일관성과 원자성을 확보했습니다. 이 과정에서 데이터 무결성을 지키는 방법을 깊이 고민하며 구현할 수 있었습니다.

    🥳 프로젝트 결과 및 성과

    EC2 Ubuntu 환경에 서비스를 배포하며 Linux의 기본적인 명령어에 익숙해졌고, Docker를 통해 컨테이너들을 관리하며 Docker를 이해할 수 있었습니다.

    또한 Jenkins를 활용해 CI/CD 파이프라인을 구축함으로써 배포 자동화에 대한 개념을 익히고 이를 직접 구현할 수 있는 역량을 갖추게 되었습니다.

    📝 사용 이유

  • Elasticsearch: 60,000개의 약품명 데이터를 빠르고 정확하게 검색할 수 있도록 하기 위해 Elasticsearch를 사용했습니다. 강력한 검색 및 자동 완성 기능을 제공해 사용자가 약품명을 입력할 때 신속하게 결과를 반환할 수 있었습니다.
  • WebRTC: 실시간 화상 통신 기능을 구현하기 위해 WebRTC를 사용했습니다. 이 기술은 브라우저 간의 비디오 및 오디오 스트리밍을 가능하게 하며, 빠른 데이터 전송과 안정적인 연결을 보장합니다. 특히, 화상 신고 기능에 적합한 성능을 제공했습니다.
  • OpenVidu: WebRTC를 기반으로 하는 OpenVidu를 사용하여 화상 통신을 쉽게 관리하고 제어할 수 있었습니다. OpenVidu는 복잡한 WebRTC API를 보다 단순화해주어 화상 신고 기능의 개발 시간을 단축시키고, 효율적인 멀티미디어 관리가 가능하게 해주었습니다.
  • PWA (Progressive Web App): 앱 설치 없이도 웹 환경에서 모바일 앱과 유사한 경험을 제공하기 위해 PWA를 도입했습니다. 사용자는 브라우저에서 바로 앱을 사용할 수 있고, 오프라인 지원과 푸시 알림 기능 등 앱 수준의 사용자 경험을 누릴 수 있었습니다.

    • ⚠️겪게된 어려움과 해결 방법

    1. Docker 환경에서 Elasticsearch 설정

    문제점: Docker를 처음 사용해 Elasticsearch를 구동하는 과정에서 많은 오류가 발생했습니다. 특히 터미널 환경에서 curl 명령어로 작업하는 것이 익숙하지 않아 비효율적이었고, Kibana Dev Tools를 활용하려 했지만 Docker와의 연결 설정이 쉽지 않았습니다. Docker의 개념조차 생소한 상황에서 반복되는 오류를 마주하며 좌절감을 느꼈습니다.

    해결 방법: 다양한 블로그와 기술 문서를 찾아 읽어가며 계속해서 방법을 찾던 중, ELK를 미리 설정해두고 컨테이너로 실행할 수 있는 GitHub 레포지토리를 발견했습니다. 이를 기반으로 설정을 수정하고, 마침내 Elasticsearch와 Kibana를 성공적으로 실행할 수 있었습니다.

    2. 약 60,000개 약품명 데이터 적재

    문제점: 방대한 데이터를 효율적으로 적재하는 일은 또 다른 어려움이었습니다. 단순히 인덱스를 만드는 것은 쉬웠지만, 대량의 데이터를 인덱스에 한번에 저장하는 데 어려움을 겪었습니다.

    해결 방법: 검색을 통해 Logstash의 파이프라인 기능을 통해 csv 파일의 데이터를 적재할 수 있다는 것을 알게 되었고, 관련 자료를 찾아 여러 번 시도했습니다. Logstash pipeline 설정이 생소해 하루 종일 실패만 거듭했지만, 주말에도 시간을 아낌없이 투자해가며 시도했고 수많은 시행착오 끝에 마침내 약 60,000개의 약품명 데이터를 Elasticsearch에 성공적으로 저장할 수 있었습니다.

    3. 검색 최적화

    문제점: 약품명 검색 기능을 최적화하는 것은 특히 어렵게 느껴졌습니다. 사용자가 약품명의 일부만 입력했을 때 정확한 결과를 반환하기 위해 Nori 토크나이저를 사용했지만, 약품명들은 단순히 영어 발음을 한국어로 표기한 것이 많았기 때문에 의도대로 분할되지 않았습니다. 특히 사용자가 약품명의 약품명의 첫 두세 글자만 입력했을 때는 검색 성능이 더욱 떨어졌습니다.

    해결 방법: 더 나은 방법을 찾기 위해 공식 문서와 여러 블로그를 철저히 조사했습니다. 그러다 Edge NGram 토크나이저를 발견했습니다. 이 토크나이저는 텍스트를 짧은 단위로 분리해 효율적으로 검색할 수 있도록 도와줍니다. 예를 들어, "아세트아미노펜"이라는 약품명을 "아", "아세", "아세트"와 같은 형태로 분리해 검색 성능을 극대화할 수 있었습니다. 이렇게 여러 필터를 실험하고 적용하며 최적의 성능을 끌어내기 위해 많은 노력을 기울였습니다.

    결과

    여러 어려움을 겪었지만 제가 맡은 임무를 끝까지 포기하지 않고 책임감있게 노력하여 ELK를 활용한 약품명 자동완성 기능을 성공적으로 구현했습니다. MySQL에 적재해 약품명을 통해 데이터를 조회하는 쿼리와 비교했을때 3배의 속도 차이를 낼 수 있었습니다. 또한 Docker와 Docker-Compose를 사용해 컨테이너 환경에서 ELK 스택을 구축하는 방법을 배웠고, Elasticsearch를 효과적으로 활용하는 법도 익혔습니다. 기술적인 문제에 부딪힐 때마다 좌절하지 않고 최선을 다해 끝까지 해결책을 찾아냈고, 그 과정에서 기술적 성장과 큰 성취감을 느낄 수 있었습니다.

    🔍 회고

    이번 응급 신고 서비스 개발 프로젝트를 통해 사회에 공헌할 수 있는 서비스를 직접 만들 수 있어서 매우 뜻깊었습니다. 많은 인원과 함께 작업하며 협업 능력을 크게 발전시킬 수 있었던 것도 좋은 경헙이었습니다. API를 개발한 후 프론트엔드 팀이 제가 만든 API가 실제로 사용되는 과정에서, 원활한 소통의 중요성을 깊이 깨달았습니다. 서로의 작업을 이해하고 맞춰가는 과정이 생각보다 복잡했지만, 덕분에 협업 능력을 한 단계 더 성장시킬 수 있었습니다. 또한, 기술적으로도 크게 성장할 수 있었습니다. 직접 ELK 스택을 활용해 성능을 개선하고, 그 결과를 눈으로 확인할 때 느낀 성취감은 정말 뿌듯했습니다. 이 프로젝트는 기술적 성장뿐만 아니라 사회적 가치를 창출하는 경험이었고, 앞으로도 다양한 기술을 활용해 의미 있는 서비스를 개발하고 싶다는 열정을 더욱 키워주었습니다.

    2. 개발을 시작할 때 저희 팀은 역할을 백엔드와 프론트엔드로 구분하지 않고 기능별로 분담했습니다. 저는 상품 관련 기능을, 페어는 유저 기능을 먼저 구현하기로 했습니다. 그러나 상품 추가 기능은 판매자 페이지에서 이루어져야 했기 때문에 유저 기능이 필요했습니다. 처음에는 권한 없이 상품 추가 기능을 구현한 후, 페어가 유저 기능을 완성하면 다시 수정하는 방식으로 개발을 진행했는데, 이 과정에서 많은 시간을 소모하게 되었습니다. 또한, 각자 맡은 부분을 개발하고 통합하는 과정에서, 개발 당시에는 알아차리지 못했던 오류들이 통합 후에 발견되었고, 이를 수정하는 데에도 상당한 시간이 소요되었습니다.

    이후부턴 백엔드와 프론트엔드 영역으로 나누어 제가 백엔드 쪽을 맡아 프로젝트를 진행하였습니다.

    🥳 프로젝트 결과 및 성과

    이번 프로젝트는 SSAFY에서 한 학기(5개월) 동안 학습한 Spring Boot, MyBatis, MySQL, Vue.js를 실전에 적용한 첫 번째 웹 프로젝트였습니다.
    처음으로 누구의 도움 없이 팀원과 함께 기획부터 개발까지 프로젝트 전 과정을 경험하며 학습한 내용을 체계적으로 정리하고, 실제로 활용할 수 있는 능력을 키울 수 있었습니다. 또한 처음으로 팀 프로젝트를 진행하며 의사소통의 중요성을 깨달았고 협업 능력을 키울 수 있었으며 GitHub를 활용한 협업 방식을 이해하고 깃 사용법을 알게 됐습니다.

    또한 Jenkins를 활용해 CI/CD 파이프라인을 구축함으로써 배포 자동화에 대한 개념을 익히고 이를 직접 구현할 수 있는 역량을 갖추게 되었습니다.

    🔍 회고

    단순히 개발만 잘한다고 되는 것이 아니라, 설계와 기획의 중요성을 실감했습니다. 잘 작성된 설계서는 개발을 훨씬 더 원활하게 진행할 수 있게 한다는 점도 배울 수 있었습니다. 처음부터 명확한 역할 분담이 필요하다는 점도 느꼈습니다. 백엔드와 프론트엔드로 분리하지 않고 기능별로 개발할 경우, 의존성이 있는 기능에 대해 미리 구체적인 계획을 세우지 않으면 효율성이 떨어질 수 있다는 것을 알았습니다. 팀원과의 의사소통을 중요시하며 작업 영역을 명확히 정의해야 프로젝트를 더 효율적으로 진행할 수 있다는 것을 알았습니다.
